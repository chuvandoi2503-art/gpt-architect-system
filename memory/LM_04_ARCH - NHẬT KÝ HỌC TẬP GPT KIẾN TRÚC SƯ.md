# BÀI HỌC 001

## Chủ đề

GPT có thể thay thế kho lưu trữ tri thức hay không.

### Quan sát

Trong quá trình làm việc, GPT nhiều lần được yêu cầu:

- Đọc tài liệu.
- Cập nhật tài liệu.
- Xuất lại tài liệu.

### Giả thuyết ban đầu

GPT có thể đóng vai trò kho lưu trữ trung tâm.

### Kiểm chứng

Khi tài liệu dài dần:

- Xuất hiện hiện tượng rút gọn.
- Xuất hiện hiện tượng mất nội dung.
- Xuất hiện hiện tượng diễn giải lại.

### Kết luận

Giả thuyết sai.

### Bài học

GPT không phù hợp làm kho lưu trữ tri thức gốc.

# BÀI HỌC 002

## Chủ đề

### Quan sát

- Cập nhật phiên bản.
- Xuất phiên bản mới.

### Kiểm chứng

GPT không đảm bảo:

- Lịch sử thay đổi.
- Khả năng khôi phục.

### Kết luận

GPT không thay thế Git.

### Bài học

Version phải được quản lý bởi hệ thống chuyên trách.

# TỔNG KẾT HIỆN TẠI

Các bài học quan trọng nhất đã hình thành framework hiện tại:

- GPT không phải kho lưu trữ.
- GPT không phải Git.
- Cần bộ nhớ ngoài.
- Cần phân tầng bộ nhớ.
- Cần lớp ÁP DỤNG BẮT BUỘC.
- Cần quy trình QC rõ ràng.
- Cần hoàn thiện một GPT mẫu trước khi nhân bản.

Những bài học này là nguồn gốc của:

- 00
- 01
- 02
- ÁP DỤNG BẮT BUỘC
- 03A
- 03B
- 04.1

Nếu nhìn toàn bộ framework thì tôi đánh giá:

\`\`\`text

00 = Luật

01 = Hệ điều hành

02 = Danh tính

ÁP DỤNG CHUNG = Quy tắc vận hành chung

ÁP DỤNG RIÊNG = Quy tắc vận hành riêng

03A = Điều phải nhớ

04.1 = Điều đang làm

03B = Điều đã biết chắc

04 = Điều đã học được

Và đây là lần đầu tiên tôi thấy bộ khung đã khép kín. Sau khi 04 PASS, việc còn lại không phải thiết kế nữa mà là **vận hành thử bằng phiên mới** để xem framework có thực sự hoạt động hay không.

trước khi triển khai thực tế.

\## BÀI HỌC MỚI - REVIEW CHI TIẾT WORKFLOW

Quan sát:

Khi thiết kế luồng GPT → n8n → Git, nhiều rủi ro chi tiết chỉ được phát hiện khi người dùng hỏi.

Các rủi ro gồm:

\- GPT đặt sai ID.

\- GPT đặt sai tên patch.

\- n8n ghi đè file cũ.

\- Số thứ tự patch bị trùng.

\- Retry tự động có thể tốn API.

\- n8n không được tin GPT tuyệt đối.

\- Patch mới phải CREATE_ONLY, không UPDATE.

\- Người dùng phải duyệt nội dung cuối cùng.

Kết luận tạm thời:

Khi thiết kế workflow, GPT Kiến Trúc Sư phải chủ động review:

\- Naming / Đặt tên.

\- Validation / Kiểm tra hợp lệ.

\- Overwrite / Ghi đè.

\- Retry / Thử lại.

\- API Cost / Chi phí API.

\- Rollback / Khôi phục.

\- Manual Approval / Duyệt thủ công.

Trạng thái:

Bài học mới.

Chưa đưa vào 03B.

Cần kiểm chứng thêm qua các workflow tiếp theo.

## 4. CẤU TRÚC NODE ĐÃ CHỐT

### Node 1 - WEBHOOK

Mục đích:

Nhận dữ liệu POST từ bên ngoài.

Dữ liệu đầu vào gồm:

```text
GPT_ID
TARGET_ID
ACTION
CONFIDENCE
TITLE
REASON
CONTENT
```

---

### Node 2 - VALIDATE_PATCH_INPUT

Mục đích:

Kiểm tra dữ liệu đầu vào.

Code làm:

```text
Kiểm tra thiếu trường
Kiểm tra GPT_ID hợp lệ
Kiểm tra TARGET_ID hợp lệ
Kiểm tra ACTION hợp lệ
Chuẩn hóa CONFIDENCE
Chặn nếu GPT tự gửi PATCH_ID
Thêm RECEIVED_AT
```

---

### Node 3 - LIST_EXISTING_PATCHES

Mục đích:

Đọc danh sách file trong:

```text
patches/draft
```

Để biết hôm nay đã có patch số mấy.

---

### Node 4 - GENERATE_PATCH_FILE

Mục đích:

Tạo patch hoàn chỉnh.

Code làm:

```text
Lấy GPT_ID
Lấy TARGET_ID
Lấy ngày hiện tại
Đọc danh sách patch hiện có
Tìm số lớn nhất
Tăng +1
Tạo PATCH_ID
Tạo FILE_NAME
Tạo FILE_PATH
Tạo MARKDOWN
Tạo COMMIT_MESSAGE
```

Định dạng PATCH_ID đã pass:

```text
PATCH_[GPT_ID]_[TARGET_ID]_[YYYYMMDD]_[STT]
```

Ví dụ:

```text
PATCH_ARCH_WM_04_1_ARCH_20260617_009
```

---

### Node 5 - CREATE_GITHUB_FILE

Mục đích:

Tạo file patch nháp trên GitHub.

Input lấy từ node GENERATE_PATCH_FILE:

```text
FILE_PATH
MARKDOWN
COMMIT_MESSAGE
```

Quy tắc:

```text
Create file only
Không update
Không ghi đè
```

---

### Node 6 - BUILD_RESPONSE

Mục đích:

Tạo JSON phản hồi gọn.

Output:

```json
{
  "status": "PASS",
  "patch_id": "...",
  "target_id": "...",
  "file_path": "...",
  "message": "Patch draft created successfully."
}
```

---

### Node 7 - RESPOND_TO_WEBHOOK

Mục đích:

Trả kết quả về nơi gọi webhook.

Cấu hình:

```text
Respond With: First Incoming Item
```
